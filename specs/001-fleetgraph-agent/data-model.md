# FleetGraph Agent Data Model

## Overview

FleetGraph uses Ship's existing document graph and REST APIs as its runtime data source. The initial design avoids direct database access and treats persisted outputs as document-backed content wherever possible.

## Core Runtime Entities

### Agent Invocation Context

- **Purpose**: Captures where and why the agent is running.
- **Fields**:
  - `trigger_type`: `on_demand` | `scheduled` | `event`
  - `view_type`: `issue` | `week` | `project` | `program` | `person` | `workspace`
  - `document_id`: optional current document identifier
  - `actor_user_id`: optional requesting user
  - `workspace_id`
  - `time_window`: optional time boundary used for scans
  - `correlation_id`
- **Validation rules**:
  - `document_id` is required for all view-specific on-demand requests.
  - `actor_user_id` is required for user-triggered requests.
  - `workspace_id` is required for every invocation.

### Agent State Snapshot

- **Purpose**: The in-flight graph state shared across context, fetch, reasoning, and action nodes.
- **Fields**:
  - `invocation`: Agent Invocation Context
  - `context_summary`
  - `fetched_resources`
  - `detected_findings`
  - `recommended_actions`
  - `draft_outputs`
  - `approval_requirements`
  - `errors`
  - `fallback_status`
- **Validation rules**:
  - Must always preserve a complete audit trail of findings and actions proposed during the run.
  - Errors must be additive so fallback nodes can inspect prior failures.

### Agent Finding

- **Purpose**: Represents actionable drift between plan and reality.
- **Fields**:
  - `finding_id`
  - `category`: `blocker` | `stale_work` | `capacity_risk` | `planning_gap` | `slipping_scope` | `silent_project` | `missing_standup`
  - `severity`: `critical` | `high` | `medium` | `low`
  - `headline`
  - `rationale`
  - `evidence`
  - `related_documents`
  - `recommended_audience`
  - `requires_human_action`
  - `confidence`
- **Validation rules**:
  - Every finding must include evidence and a stated audience.
  - Findings without a concrete action path should be suppressed before final output.

### Recommendation

- **Purpose**: A human decision the agent wants to tee up.
- **Fields**:
  - `recommendation_id`
  - `type`: `escalate` | `reassign` | `rescope` | `approve_plan` | `review_blocker` | `publish_draft`
  - `reason`
  - `expected_impact`
  - `affected_documents`
  - `approval_status`: `not_required` | `pending` | `approved` | `rejected`
  - `proposed_payload`
- **Validation rules**:
  - Any recommendation that changes Ship data must begin as `pending`.
  - `proposed_payload` must be absent for awareness-only recommendations.

### Automated Draft

- **Purpose**: Editable content generated by the agent.
- **Fields**:
  - `draft_id`
  - `draft_type`: `standup` | `weekly_plan` | `week_summary` | `portfolio_report`
  - `title`
  - `body`
  - `source_context`
  - `status`: `draft` | `reviewed` | `published`
  - `automated_marker`
- **Validation rules**:
  - New persisted draft documents must use the initial title `Untitled`.
  - Every draft must remain visibly marked as agent-generated until a human edits or confirms it.

### Approval Gate

- **Purpose**: Tracks a human-in-the-loop checkpoint.
- **Fields**:
  - `gate_id`
  - `gate_type`: `mutation` | `notification_send` | `document_publish`
  - `decision_owner`
  - `decision_reason`
  - `status`: `pending` | `approved` | `rejected` | `expired`
  - `blocked_action_ids`
- **Validation rules**:
  - Mutation and publish gates are mandatory before any state-changing action is sent to Ship APIs.
  - Rejected gates must preserve the proposed action for audit and follow-up.

### Fallback Event

- **Purpose**: Captures degraded execution when data is missing, an upstream dependency fails, or confidence is too low.
- **Fields**:
  - `fallback_id`
  - `error_type`
  - `user_safe_message`
  - `recovery_action`
  - `retryable`
  - `logged_at`
- **Validation rules**:
  - Every fallback must have a user-safe explanation.
  - Retryable failures must be distinguishable from hard failures.

## Document Model Mapping

### Existing Ship Documents Used as Inputs

- **Issue documents**: state, priority, assignee, estimates, history, comments
- **Week documents**: ownership, approval state, assigned issues, weekly plan and retro children
- **Project documents**: backlog grouping and project-level health context
- **Program documents**: cross-project rollups
- **Person documents**: capacity, assignments, activity context

### Persisted Outputs

- **Draft outputs**: Stored as document-backed content so users can review and refine them in Ship
- **Summaries and insights**: Stored as document-backed artifacts when they must persist beyond a single run
- **Approval requests and transient findings**: May remain API response objects unless product testing shows a need for durable storage

## Relationships

- One **Agent Invocation Context** produces one **Agent State Snapshot**
- One **Agent State Snapshot** may contain many **Agent Findings**
- One **Agent Finding** may produce zero or many **Recommendations**
- One **Recommendation** may require one **Approval Gate**
- One **Agent State Snapshot** may produce zero or many **Automated Drafts**
- One **Agent State Snapshot** may produce zero or many **Fallback Events**

## State Transitions

### Recommendation Approval Status

- `not_required` -> terminal for awareness-only recommendations
- `pending` -> `approved`
- `pending` -> `rejected`
- `pending` -> `expired`

### Automated Draft Status

- `draft` -> `reviewed`
- `reviewed` -> `published`
- `draft` -> `published` only when a human explicitly confirms publication

### Approval Gate Status

- `pending` -> `approved`
- `pending` -> `rejected`
- `pending` -> `expired`
